설계 하이라이트

① 전투 엔진을 partial class로 분리

약 2,665줄로 커진 BattleManager를 5개 파일로 책임 분리 — 흐름(.Phases)과 계산(.Combat), 적 AI(.EnemyAction)를 격리해 유지보수성 확보.

② 반복 패턴을 CRTP 제네릭으로 일반화

Singleton<T>·BaseCurrency<T>·RoleCostBase<T>로 인스턴스 관리·저장·UI 갱신의 중복을 제거. 타입 안전 + 중복 인스턴스 자동 제거.

③ 데이터 주도 설계 (JSON + SO)

동료·스킬·적 정의를 JSON으로 빼내 기획자가 코드 없이 밸런싱. DB 5종이 Dictionary 캐싱·조회만 담당하고 상태는 SO가 관리.

④ 데이터로 조종하는 적 AI

가중치 랜덤 스킬 선택 + 타겟팅 매트릭스(도발·소환·독). weight·targeting 값만 바꾸면 AI 재구성. 4스킬 보스 구현.

03대표 프로젝트 ② — 바른자세 코치봇

ESP32-S3 기반 AI 자세 코치 로봇 ESP-IDF v5.5.4 · C / C++ · FreeRTOS · Xiaozhi AI · 펌웨어 & 하드웨어 브링업

기울기 센서로 앞으로 숙인 자세를 감지하면 AI 음성으로 교정해 주는 로봇. 원래 키트의 4족 보행 기능은 목적에 불필요해서 서보를 제거하고, 자세 감지·음성에 집중했습니다.

시스템 구성 — 남긴 것과 제거한 것

그림 2. 실제 시스템 = 기울기 센서 → 자세 판단 → AI 음성. 보행은 목적에 맞춰 걷어냈다.

기울기 센서로 자세 감지

사용자가 앞으로 숙였는지를 기울기 센서로 판정해, 자세가 무너지면 코칭 이벤트를 발생시키는 핵심 입력을 구성.

I2S 오디오 입출력

i2s_std로 INMP441 마이크·MAX98357A 스피커를 구동. 포트(I2S_NUM_0) 하나를 마이크↔스피커로 번갈아 사용해 충돌 회피(06_mic_reaction_test).

AI 프레임워크 이식 · 크로스빌드

Xiaozhi AI 펌웨어(디바이스 상태머신·mcp_server·OTA)를 Windows 전용 가이드를 macOS로 이식해 ESP-IDF v5.5.4로 빌드 성공.

불필요한 하드웨어 제거 (설계 판단)

키트에 딸린 서보·4족 보행이 목적과 무관함을 확인하고 과감히 제거. 단계별 브링업으로 검증 후 남길 것·버릴 것을 구분한 결정.

05집중 스킬 심층 정리

여러 프로젝트를 관통하는 세 가지 집중 스킬과, 그것을 뒷받침하는 실제 근거를 정리했습니다.

① 대규모 코드를 “나누는” 아키텍처 감각  — 가장 깊은 역량

  • 모듈 분리: 2,665줄 전투 시스템을 partial class 5개로, 비대한 데이터 클래스를 .Hp/.Skill/.Stress로 분할.
  • 중복 일반화: 싱글톤·재화·스택의 공통 로직을 CRTP 제네릭 베이스로 추출.
  • 관심사 분리: 정의 데이터(JSON)와 상태(SO), 로직과 표현(이벤트 기반)을 분리.
근거INSPECTOR — BattleManager.*, Singleton<T>, BaseCurrency<T>, 5종 JSON 데이터베이스

② 하드웨어를 바닥부터 올리는 임베디드 브링업  — 두 번째 집중 영역

  • 단계적 검증: 보드→OLED→스피커→마이크→반응→AI를 독립 프로젝트로 하나씩 확인 후 통합.
  • 저수준 통신: i2s_std로 INMP441 마이크·MAX98357A 스피커를 직접 구동(포트 번갈아 사용).
  • 목적 중심 정리: 키트의 서보·4족 보행이 불필요함을 확인하고 제거 — 남길 것·버릴 것을 구분.
  • 환경 정복: Windows 전용 빌드를 macOS로 이식, 실패 함정을 스크립트·문서로 재현 가능하게 정리.

06개발 방식 — AI 협업 개발(바이브 코딩)

이 프로젝트들의 상당 부분은 AI 협업 개발(바이브 코딩)으로 구현했습니다. 도구가 코드를 빠르게 써 주는 만큼, 제 역할은 무엇을 왜 만들지 정하고, 결과를 검증·통합·디버깅하는 데 있었습니다. 이 점을 숨기지 않고, 하나의 실전 역량이자 학습의 출발점으로 정리합니다.

역할 분담 — 내가 한 것 vs AI가 가속한 것

작업주도내용
요구 정의 · 문제 분해본인무엇을·왜 만들지, 완료 기준 설정

사용한 AI 협업 도구

Claude Code · Unity MCP(에디터 자동 조작) · ESP-IDF 워크플로우 · execute_code(애니메이터·NavMesh 빌드) · 플래시/데모 Bash 자동화.

나의 바이브 코딩 루프

① MCP로 현재 상태 확인 → ② 작업 이해를 내 말로 정리 → ③ 확인·승인 → ④ 실행 후 직접 검증·수정. 이해하지 못한 채 복붙하지 않는 것을 원칙으로 합니다.

이 방식을 어떻게 바라보는가 (정직하게) AI 협업 개발은 도구 오케스트레이션 · 프롬프트 설계 · MCP 자동화라는 실전 역량이라고 생각합니다. 동시에, 이렇게 빠르게 “만들 수 있게” 된 것과 “밑바닥까지 이해한 것” 사이엔 아직 간극이 있음을 인정합니다. 그 간극을 메우기 위한 학습 계획을 다음 장에 정리했습니다.
